Architecture Overview

High-level architecture of Fleet Credits Core

System Architecture

Fleet Credits Core is built on a modular architecture designed for scalability, privacy, and contribution-based rewards.

Core Components

Application Layer

  • fleetcreditsd - Core daemon
  • fleetcredits-qt - Qt GUI client
  • fleetcredits-cli - Command-line interface
  • fleetcredits-tx - Transaction utility

Consensus Layer

  • Scrypt PoW - Proof-of-Work consensus
  • Block Validation - Transaction and block verification
  • Contribution System - Contribution transaction processing
  • Reward Distribution - Block reward calculation with bonuses

Transaction Layer

  • UTXO Model - Unspent Transaction Output model
  • Standard Transactions - P2PKH, P2SH, etc.
  • Contribution Transactions - 8 contribution types
  • MWEB Transactions - Privacy-enhanced transactions

Privacy Layer (MWEB)

  • Pedersen Commitments - Amount hiding
  • Range Proofs - Non-negative value proofs
  • Cut-through - Blockchain bloat reduction
  • Peg-in/Peg-out - Atomic swaps with main chain

Verification Layer

  • Oracle Nodes - Decentralized verification
  • Human Validators - Reputation-based validation
  • Verification Records - Contribution verification tracking
  • Consensus Mechanism - 3-of-5 oracle consensus

Network Layer

  • P2P Protocol - Peer-to-peer communication
  • RPC Server - JSON-RPC API
  • Network Parameters - Mainnet, testnet, regtest

Storage Layer

  • Blockchain Database - Block and UTXO storage
  • Wallet Database - Wallet persistence
  • Mempool - Unconfirmed transaction pool

Data Flow

Transaction Submission Flow

  1. User creates transaction via RPC, CLI, or GUI
  2. Transaction validated against consensus rules
  3. Transaction added to mempool
  4. Miner includes transaction in block
  5. Block validated and added to blockchain
  6. Rewards distributed (base + contribution bonuses)

Contribution Flow

  1. User submits contribution via submitcontribution RPC
  2. Contribution transaction created with proof data
  3. If MWEB required (ETHICAL_REVIEW), route to MWEB extension block
  4. Verification record created
  5. Oracles vote on contribution validity
  6. Consensus achieved (3-of-5 approval)
  7. Bonus multiplier applied to block reward
  8. Reward distributed to miner

Key Design Decisions

  • Fixed Block Reward: 1.00000069 FC per block (no halving) to support post-scarcity economics
  • Zero-Fee Micro-transactions: Transactions < 0.01 FC have zero fees
  • Contribution Bonuses: Dynamic reward multipliers based on verified contributions
  • MWEB Integration: Optional privacy for standard transactions, mandatory for sensitive contributions
  • Oracle Consensus: Decentralized verification with 3-of-5 consensus threshold
  • 60-Second Block Time: Faster confirmations compared to Bitcoin
