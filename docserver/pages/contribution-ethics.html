Ethics & Value Judgment

The Challenge of Moral Judgment

Fleet Credits faces a fundamental question: How do we determine what constitutes a "good" vs "evil" deed or contribution? This is especially critical for subjective contribution types like CHARITABLE_ACT and ETHICAL_REVIEW, where moral value judgments are required.

Files: src/primitives/verification.h, src/primitives/verification.cpp

Core Principle: Fleet Credits does not encode a single moral framework. Instead, it uses decentralized oracle consensus to reflect the collective values of the community. What is "good" is determined by community consensus, not by a central authority.

Why This Matters

Without clear criteria, the system could:

  • Reward harmful or manipulative contributions
  • Exclude legitimate contributions due to bias
  • Be exploited by malicious actors
  • Fail to incentivize truly valuable contributions

Therefore, Fleet Credits implements a multi-layered approach combining objective criteria, community consensus, and safeguards against abuse.

Verification Framework

1. Objective Criteria (Hard Requirements)

Before any moral judgment, contributions must meet objective technical requirements:

For All Contributions

  • Valid Proof: Evidence must be verifiable and authentic
  • Non-Duplicate: Cannot be a replay of previous contribution
  • Proper Format: Must match contribution type structure
  • Minimum Thresholds: Must meet size/quality minimums
  • Authentic Source: Contributor identity must be verified

For CHARITABLE_ACT

  • Verifiable Receipt: Donation receipt or service certificate
  • Minimum Impact: Must benefit identifiable recipients
  • Non-Self-Serving: Cannot benefit the contributor directly
  • Public Benefit: Must serve community or public good

For ETHICAL_REVIEW

  • Substantive Analysis: ‚â•100 words, non-boilerplate
  • Relevant Context: Must address actual AI system or proposal
  • MWEB Privacy: Mandatory for sensitive reviews
  • Ethical Framework: Must reference recognized ethical principles

2. Oracle Consensus (Value Judgment)

Once objective criteria are met, oracles make value judgments based on:

Oracle Evaluation Criteria

Oracles evaluate contributions using these dimensions:

DimensionCHARITABLE_ACTETHICAL_REVIEW
Impact How many people benefit? How significant is the benefit? Does the review address real risks? Will it prevent harm?
Authenticity Is the charitable act genuine? Is it actually altruistic? Is the ethical analysis sincere? Does it show genuine concern?
Alignment Does it align with community values? Does it serve public good? Are the ethical principles sound? Do they align with human dignity?
Verifiability Can the impact be verified? Is there evidence of benefit? Can the ethical concerns be validated? Are they reasonable?
Prevention of Harm Does it avoid causing harm? Does it respect rights? Does it identify potential harms? Does it propose safeguards?

Oracle Decision-Making Process

Step-by-Step Evaluation

When an oracle evaluates a contribution, they follow this process:

Step 1: Objective Verification

  1. Verify proof data exists and is authentic
  2. Check contribution meets minimum technical requirements
  3. Verify contributor identity and reputation
  4. Check for replay attacks or duplicates

Step 2: Value Assessment

Oracles assess the contribution's value using:

  • Impact Score: How many people benefit? How significant?
  • Authenticity Score: Is this genuine or manipulative?
  • Alignment Score: Does it align with community values?
  • Harm Prevention Score: Does it prevent harm?

Step 3: Consensus Building

Multiple oracles vote independently:

  • Approve: Contribution is valuable and should be rewarded
  • Reject: Contribution fails to meet standards
  • Abstain: Oracle cannot make a determination

Step 4: Threshold Check

For contributions requiring oracle consensus:

  • Minimum Votes: At least 3 of 5 oracles must vote
  • Approval Threshold: ‚â•60% approval rate (3 of 5)
  • Consensus: If threshold met, contribution is verified

Oracle Diversity

To prevent bias, oracle pools are selected for diversity:

  • Geographic Diversity: Oracles from different regions
  • Cultural Diversity: Oracles from different cultural backgrounds
  • Expertise Diversity: Oracles with different areas of expertise
  • Reputation-Based: Oracles with proven track records

Philosophical Framework

Fleet Credits doesn't enforce a single moral philosophy, but contributions are evaluated against principles that reflect broad consensus:

Core Principles

1. Human Dignity

  • Contributions must respect human dignity and rights
  • Contributions that dehumanize or harm are rejected
  • Examples: Supporting human rights organizations ‚úÖ, supporting hate groups ‚ùå

2. Public Good

  • Contributions should benefit the public or community
  • Self-serving contributions are evaluated more strictly
  • Examples: Open-source software ‚úÖ, personal business promotion ‚ùå

3. Non-Maleficence (Do No Harm)

  • Contributions must not cause harm
  • Harm prevention is prioritized over benefit
  • Examples: Ethical reviews preventing AI bias ‚úÖ, reviews promoting discrimination ‚ùå

4. Verifiability

  • Claims must be verifiable
  • Evidence must be authentic and accessible
  • Examples: Verified donation receipts ‚úÖ, unverifiable claims ‚ùå

5. Authenticity

  • Contributions must be genuine
  • Manipulative or fraudulent contributions are rejected
  • Examples: Sincere charitable acts ‚úÖ, fake receipts ‚ùå

Cultural Relativism vs Universal Values

Fleet Credits balances these approaches:

  • Universal Values: Core principles that apply across cultures (human dignity, non-harm)
  • Cultural Context: Oracles consider cultural context when evaluating contributions
  • Community Consensus: What is "good" evolves with community consensus
  • Governance: Community can vote to update evaluation criteria

Safeguards Against Abuse

1. Oracle Selection

  • Stake Requirements: 500,000 FC locked prevents casual participation
  • Reputation Tracking: Oracle performance is tracked and affects selection
  • Community Governance: Community votes on oracle selection
  • Rotation: Oracles can be rotated to prevent stagnation

2. Slashing Mechanisms

Oracles face economic penalties for:

  • Consensus Violations: Voting against clear consensus
  • Pattern Detection: Systematic approval of fraudulent contributions
  • Collusion: Coordinated voting with other oracles
  • Malicious Behavior: Deliberately approving harmful contributions

3. Appeal Process

Contributors can appeal rejected contributions:

  • Provide additional evidence
  • Request re-evaluation by different oracle pool
  • Community can vote to override oracle decision

4. Transparency

  • Oracle votes are recorded on-chain
  • Contribution evaluations are publicly auditable
  • Patterns of abuse can be detected and punished

5. Minimum Standards

Even with oracle consensus, contributions must meet minimum standards:

  • Cannot violate fundamental human rights
  • Cannot support illegal activities
  • Cannot cause demonstrable harm
  • Must have verifiable evidence

Edge Cases & Controversial Contributions

How System Handles Controversy

Politically Sensitive Contributions

Challenge: What if a contribution is legitimate but politically controversial?

Approach:

  • Oracles evaluate based on impact and harm prevention, not political alignment
  • Contributions that serve public good are evaluated positively
  • Contributions that cause harm are rejected regardless of politics
  • Community governance can override oracle decisions

Religious Contributions

Challenge: Should religious charitable acts be rewarded?

Approach:

  • Evaluated based on public benefit, not religious content
  • Religious organizations providing public services (food, shelter, education) are treated equally
  • No discrimination based on religious affiliation
  • Must meet same standards as secular contributions

AI Ethics Disagreements

Challenge: What if ethicists disagree on AI safety?

Approach:

  • Multiple valid perspectives can coexist
  • Oracles evaluate based on reasoning quality, not specific conclusions
  • Reviews that identify real risks are rewarded, even if solutions differ
  • Reviews promoting harm are rejected regardless of academic credentials

Competing Values

Challenge: What if values conflict (e.g., privacy vs security)?

Approach:

  • Oracles evaluate based on harm prevention and public good
  • Contributions proposing balanced solutions are rewarded
  • Contributions promoting extreme positions may be evaluated more strictly
  • Community can vote to clarify evaluation criteria

Evolution of Standards

Community Governance

Evaluation standards evolve through community governance:

  • Proposals: Community members can propose changes to evaluation criteria
  • Voting: FC holders vote on proposal acceptance
  • Implementation: Approved criteria are encoded into oracle evaluation
  • Review: Criteria are periodically reviewed and updated

Historical Context

What is considered "good" may change over time:

  • Early network: Focus on establishing basic standards
  • Growth phase: Refining criteria based on experience
  • Mature network: Stable consensus on core values
  • Evolution: Adapting to new challenges and contexts

Examples of Evolving Standards

  • Environmental Impact: May become more important over time
  • AI Safety: Standards may tighten as AI capabilities grow
  • Privacy: May become stricter as surveillance increases
  • Inclusion: May prioritize contributions from underrepresented groups

Implementation Details

Oracle Vote Structure

enum OracleVoteChoice {
    VOTE_APPROVE = 0x01,
    VOTE_REJECT  = 0x02,
    VOTE_ABSTAIN = 0x03
};

class COracleVote {
    CPubKey oracle_pubkey;
    OracleVoteChoice vote_choice;
    uint256 contribution_hash;
    std::string rationale;  // Optional explanation
    uint256 signature;      // Oracle signature
};

Consensus Calculation

bool VerifyOracleConsensus(const std::vector& votes) {
    uint32_t approve_count = 0;
    uint32_t total_votes = 0;
    
    for (const auto& vote : votes) {
        if (vote.vote_choice == VOTE_ABSTAIN) continue;
        total_votes++;
        if (vote.vote_choice == VOTE_APPROVE) approve_count++;
    }
    
    // Need at least 3 votes
    if (total_votes < 3) return false;
    
    // Need ‚â•60% approval (3 of 5 minimum)
    return (approve_count * 100 / total_votes) >= 60;
}

Slashing Logic

bool ShouldSlashOracle(const COracle& oracle, const std::vector& consensus) {
    // Slash if oracle consistently votes against consensus
    uint32_t violations = count_violations(oracle.votes, consensus);
    if (violations > oracle_violation_threshold) return true;
    
    // Slash if oracle approves fraudulent contributions
    if (detected_fraud_count(oracle) > fraud_threshold) return true;
    
    return false;
}

Conclusion

Fleet Credits addresses the challenge of moral judgment through:

  • Decentralized Consensus: Community values determine what is "good"
  • Objective Criteria: Hard requirements prevent obvious abuse
  • Economic Incentives: Oracles stake value to ensure honest evaluation
  • Transparency: All decisions are auditable on-chain
  • Evolution: Standards evolve through community governance

This system does not eliminate moral judgment‚Äîit decentralizes it, making it reflective of community consensus rather than a single authority. As the community grows and evolves, so too will the standards for what constitutes "good" contributions.

Philosophy: Fleet Credits recognizes that morality is complex and context-dependent. Rather than encoding a single moral framework, we create a system where community consensus determines values, with safeguards to prevent abuse and ensure contributions genuinely serve public good.
