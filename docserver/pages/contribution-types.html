Contribution Types

The Concept

Fleet Credits rewards contributors for verified community contributions that create value. In an era of AI-driven automation displacing traditional jobs, FC creates economic opportunities for human validation, verification, and input processing‚Äîessential for training, refining, and ethically guiding AI systems.

Philosophy: This fosters a symbiotic human-AI economy where humans earn FC for irreplaceable oversight, ensuring AI benefits society equitably. Contributions are not just rewarded‚Äîthey're the foundation of a post-scarcity economy where value derives from utility and impact, not speculation.

How It Works

  1. Submit: Contributor submits their work via submitcontribution RPC
  2. Embed: Contribution data is embedded in transaction via OP_RETURN
  3. Verify: Automated verification or oracle consensus validates the contribution
  4. Reward: Block reward includes bonus based on contribution level and verification

Verification Methods

  • Automated: Code contributions (GitHub API), AI validation (accuracy scores ‚â•0.8)
  • Oracle Consensus: Charitable acts, ethical reviews (3-of-5 oracle votes required)
  • Human Validators: Reputation-gated pools for batch verification

All 8 Contribution Types

Fleet Credits supports 8 types of contributions, each with specific verification requirements and reward structures.

1. CODE_CONTRIBUTION (0x01)

Description: Open-source code commits to public repositories (GitHub, GitLab, etc.)

Concept: Rewards developers for contributing to open-source projects, fostering innovation and collaboration.

Proof Data

  • GitHub commit hash
  • Repository URL
  • Commit metadata (lines changed, language, etc.)

Verification Requirements

  • GitHub API verification: Commit exists and belongs to repo
  • Minimum 10 lines changed
  • Code passes linting checks
  • Account age >30 days (prevents spam)
  • Replay prevention: Commit hash not used before

Class Structure

class CCodeContributionTransaction : public CContributionTransaction {
    std::string repo_url;
    uint256 commit_hash;
    uint32_t lines_changed;      // >10 required
    std::string language;
    bool passes_lint;
    uint32_t account_age_days;   // >30 required
};

Verification Process

verify_code_submission(tx) ‚Üí bool {
    commit = github_api.get_commit(tx.proof_data.commit_hash)
    return commit.lines_changed > 10 &&
           commit.passes_lint &&
           github_api.account_age(tx.contributor) > 30_days &&
           !is_replay_attack(tx.commit_hash)
}

Bonus Level

LOW to HIGH based on code quality, impact, and repository significance.

MWEB Required

No (optional for privacy)

Example Use Case

A developer fixes a critical bug in a popular open-source library. The commit has 50 lines changed, passes all linting, and the GitHub account is 2 years old. This gets verified automatically and receives a HIGH bonus level.

2. CHARITABLE_ACT (0x02)

Description: Verified charitable donations or community service

Concept: Rewards altruistic actions that benefit the community, incentivizing public goods and social impact.

Proof Data

  • Receipt hash (donation receipt, service certificate)
  • Attestation or verification link
  • Description of charitable act

Verification Requirements

  • Oracle Consensus Required: 3-of-5 oracle votes
  • Minimum 60% approval rate
  • Verifiable proof (receipt, certificate, etc.)
  • Oracle reputation-based selection

Verification Process

verify_charitable_act(tx, oracle_votes) ‚Üí bool {
    approve_count = count_approve_votes(oracle_votes)
    total_votes = count_non_abstain_votes(oracle_votes)
    
    if total_votes < 3:
        return false  // Need at least 3 votes
    
    approval_rate = approve_count / total_votes
    return approval_rate >= 0.6  // 60% threshold (3 of 5)
}

Bonus Level

MEDIUM to HIGH based on impact and oracle consensus strength.

MWEB Required

No (optional for privacy-sensitive donations)

Example Use Case

A contributor donates 10,000 FC to a verified disaster relief fund. They submit a receipt hash. Five oracles review the submission, with 4 approving and 1 rejecting. The 80% approval rate exceeds the 60% threshold, so the contribution is verified and receives a HIGH bonus level.

3. CREATIVE_WORK (0x03)

Description: Art, writing, music, or other creative content published on IPFS

Concept: Rewards artists and creators for sharing their work, fostering a vibrant creative economy.

Proof Data

  • IPFS hash of the creative work
  • Work metadata (title, description, type)
  • Creator attribution

Verification Requirements

  • IPFS hash resolves to valid content
  • Content is original (not duplicate)
  • Community validation or oracle consensus
  • Minimum quality threshold (optional)

Bonus Level

LOW to MEDIUM based on community engagement and quality.

MWEB Required

No

Example Use Case

An artist creates a digital painting, uploads it to IPFS, and submits the IPFS hash. The content is verified to exist and be original. Community members upvote it, resulting in a MEDIUM bonus level.

4. EDUCATIONAL_CONTENT (0x04)

Description: Teaching materials, tutorials, courses, or educational resources

Concept: Rewards educators for creating and sharing knowledge, democratizing education.

Proof Data

  • IPFS hash or URL to educational content
  • Content metadata (subject, level, format)
  • Engagement metrics (views, completion rate)

Verification Requirements

  • Content accessible and valid
  • Quiz responses show 80%+ accuracy (if quiz included)
  • Minimum engagement metrics
  • Quality checks (non-spam, educational value)

Verification Process

verify_educational_content(tx) ‚Üí bool {
    if has_quiz(tx.proof_data):
        quiz_result = verify_quiz(tx.proof_data)
        if quiz_result.score < 0.8:
            return false
    
    engagement = get_engagement_metrics(tx.proof_data)
    return engagement.meets_minimum() && is_educational(tx.proof_data)
}

Bonus Level

LOW to MEDIUM based on engagement and educational value.

MWEB Required

No

Example Use Case

A teacher creates a comprehensive tutorial on blockchain basics, uploads it to IPFS, and includes a quiz. Students complete the quiz with an average score of 85%. The content receives engagement metrics above the threshold and gets verified with a MEDIUM bonus level.

5. CODE_MENTORSHIP (0x05)

Description: Code reviews and mentorship between mentor and mentee

Concept: Creates a learning ecosystem where experienced developers mentor novices, building community and skills.

Proof Data

  • Mentor and mentee public keys
  • Feedback hash (mentor's review)
  • Improvement commit hash (mentee's improvements)
  • Approval status (APPROVED, REJECTED, NEEDS_WORK)
  • Both mentor and mentee signatures

Verification Requirements

  • Both mentor and mentee identities verified (DID)
  • Feedback is substantial (>50 words, non-boilerplate)
  • Both signatures valid
  • Random 10% audit via oracle consensus
  • Approval status indicates legitimate review

Class Structure

class CMentorshipTransaction : public CContributionTransaction {
    CPubKey mentor;
    CPubKey mentee;
    std::vector mentor_signature;
    std::vector mentee_signature;
    ApprovalStatus approval_status;  // APPROVED, REJECTED, NEEDS_WORK
    uint256 feedback_hash;
    uint256 improvement_commit_hash;
};

Verification Process

verify_mentorship(tx, oracles) ‚Üí bool {
    if length(tx.proof_data.feedback) < 50 ||
       is_boilerplate(tx.proof_data.feedback) ||
       !identity_verified(tx.mentor) ||
       !identity_verified(tx.mentee):
        return false
    
    if random_audit_trigger():  // 10% chance
        votes = select_random_oracles(oracles, count=5)
        oracle_votes = [oracle.verify(tx.proof_data) for oracle in votes]
        return sum(approve_votes) >= (5 * 3 / 5)  // 3 of 5
    
    return true
}

Reward Structure

  • Mentees: 1000 FC base + 500 FC for improvements
  • Mentors: 2000 FC per review + 1.10x‚Äì1.15x for complexity
  • Milestones: 5 contributions = 1.10x bonus, 10 contributions = 1.15x bonus

Bonus Level

MEDIUM to HIGH based on feedback quality and improvement impact.

MWEB Required

No

Example Use Case

A novice developer ("vibe coder") submits code for review. An experienced developer ("real coder") reviews it, provides detailed feedback (200 words), and marks it as NEEDS_WORK. The mentee improves the code based on feedback and resubmits. Both receive rewards: mentor gets 2200 FC (2000 base + 10% complexity bonus), mentee gets 1500 FC (1000 base + 500 improvement).

6. AI_VALIDATION (0x06)

Description: Human validation of AI model outputs

Concept: Critical for the AI era‚Äîhumans verify AI-generated content, code, or decisions, ensuring accuracy and preventing hallucinations. This creates scalable job opportunities as AI adoption increases.

Proof Data

  • AI model ID
  • Accuracy score (‚â•0.8 required)
  • Validation report hash (IPFS)
  • Human override hash (if human corrected AI errors)
  • Unique nonce (prevents replay attacks)

Verification Requirements

  • Accuracy score ‚â•0.8 (80% threshold)
  • Validation report exists and is valid
  • Nonce prevents duplicate validations
  • Human override verification (if applicable)

Class Structure

class CAIValidationTransaction : public CContributionTransaction {
    std::string ai_model_id;
    float accuracy_score;  // >= 0.8 required
    uint256 validation_report_hash;
    uint256 human_override_hash;
    bool human_override_verified;
};

Verification Process

verify_ai_validation(tx) ‚Üí bool {
    if tx.accuracy_score < 0.8:
        return false
    
    if !verify_report_exists(tx.validation_report_hash):
        return false
    
    if tx.human_override_verified:
        if !verify_human_override(tx.human_override_hash):
            return false
    
    if is_replay_attack(tx.nonce):
        return false
    
    return true
}

Bonus Level

CRITICAL with 1.25x multiplier (higher than standard CRITICAL 1.20x) because AI validation is essential for safe AI deployment.

MWEB Required

No (optional for privacy-sensitive validations)

Example Use Case

An AI model generates code for a function. A human validator reviews the code, finds it's 92% accurate, corrects 2 bugs, and submits a validation report. The accuracy score of 0.92 exceeds the 0.8 threshold, and the human override is verified. The validator receives rewards with a 1.25x multiplier.

Future Vision

As AI adoption increases, AI_VALIDATION becomes a scalable job market. Humans earn FC by validating AI outputs in various domains: code review, content moderation, medical diagnosis assistance, legal document review, etc.

7. DATA_LABELING (0x07)

Description: Dataset labeling with inter-annotator agreement for AI training

Concept: Rewards crowdsourced data labeling, essential for training accurate AI models. Uses consensus mechanisms to ensure quality.

Proof Data

  • Dataset ID
  • Labels hash (IPFS hash of labeled data)
  • Labels count
  • Inter-annotator agreement score (‚â•0.7 required)
  • Annotator count (‚â•3 required)

Verification Requirements

  • Minimum 3 annotators (for consensus)
  • Inter-annotator agreement ‚â•0.7 (70% consensus)
  • Labels hash points to valid IPFS data
  • Quality checks (labels are accurate and consistent)

Class Structure

class CDataLabelingTransaction : public CContributionTransaction {
    std::string dataset_id;
    uint32_t labels_count;
    uint256 labels_hash;  // IPFS hash
    float inter_annotator_agreement;  // >= 0.7 required
    uint32_t annotator_count;  // >= 3 required
};

Verification Process

verify_data_labeling(tx) ‚Üí bool {
    if tx.annotator_count < 3:
        return false
    
    if tx.inter_annotator_agreement < 0.7:
        return false
    
    if !verify_labels_hash_exists(tx.labels_hash):
        return false
    
    // Verify agreement calculation
    agreement = calculate_consensus_score(tx.labels_hash, tx.annotator_count)
    return agreement >= 0.7
}

Bonus Level

MEDIUM to HIGH based on agreement score and dataset importance.

MWEB Required

No

Example Use Case

Five annotators label images for a computer vision dataset. Their labels show 75% inter-annotator agreement (above the 0.7 threshold). The labeled data is uploaded to IPFS, and the hash is submitted. All five annotators receive rewards with a MEDIUM bonus level.

8. ETHICAL_REVIEW (0x08)

Description: Ethical review of AI systems or proposals

Concept: Ensures AI systems are safe, unbiased, and aligned with human values. Critical for responsible AI deployment.

Proof Data

  • Review report hash (IPFS)
  • Ethical rationale (‚â•100 words required)
  • Rationale length
  • Oracle consensus status

Verification Requirements

  • Rationale ‚â•100 words (substantive review)
  • Oracle consensus achieved (3-of-5 votes)
  • Review report exists and is valid
  • MWEB mandatory for privacy (sensitive reviews)

Class Structure

class CEthicalReviewTransaction : public CContributionTransaction {
    uint256 rationale_hash;  // >= 100 words required
    uint64_t rationale_length;
    bool oracle_consensus_achieved;
};

Verification Process

verify_ethical_review(tx, oracle_votes) ‚Üí bool {
    if tx.rationale_length < 100:
        return false
    
    if !tx.requires_mweb:
        return false  // MWEB mandatory
    
    if !verify_oracle_consensus(tx, oracle_votes):
        return false
    
    return true
}

Bonus Level

HIGH to CRITICAL based on review depth and impact.

MWEB Required

Yes (mandatory) for privacy‚Äîethical reviews may contain sensitive information about AI systems.

Example Use Case

An ethicist reviews an AI hiring system for bias. They write a 500-word analysis identifying potential discrimination, submit it via MWEB for privacy, and five oracles review it. Four approve, one rejects, achieving 80% consensus (above the 60% threshold). The review is verified and receives a CRITICAL bonus level.

Technical Implementation

Transaction Embedding

Contributions are embedded in standard transactions using OP_RETURN outputs:

// Contribution data is serialized and embedded in OP_RETURN
CTransaction tx;
CTxOut contrib_out;
contrib_out.scriptPubKey = CScript() << OP_RETURN << contribution_data;
tx.vout.push_back(contrib_out);

Base Structure

class CContributionTransaction {
    uint256 tx_id;
    CPubKey contributor;
    ContributionType contrib_type;
    CProofData proof_data;
    uint64_t timestamp;
    uint32_t bonus_level;
    bool requires_mweb;
    std::vector signature;
};

Proof Data Structure

class CProofData {
    uint256 hash;
    ContributionType contrib_type;
    std::vector evidence;  // IPFS hash, GitHub commit, etc.
    std::vector nonce;      // Unique per AI validation
    uint64_t timestamp;
    std::vector metadata;
};

Serialization

All contribution types use custom serialization via ADD_SERIALIZE_METHODS macro, ensuring compatibility with blockchain storage and network transmission.

Reward Integration

base_reward = 10000 * COIN
bonus_multiplier = GetBonusMultiplier(highest_bonus_level_in_block, contrib_type)
