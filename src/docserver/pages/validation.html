Validation System

Transaction and Block Validation

Fleet Credits validates transactions and blocks before adding them to the chain.

Files: src/validation.cpp, src/consensus/validation.h, src/chainparams.cpp

Transaction Validation

All transactions are validated before inclusion in blocks:

  • Script Verification: Executes and validates scriptSig and scriptPubKey
  • Fee Calculation: Zero fees for transactions <1000 FC, scalable fees otherwise
  • UTXO Availability: Ensures all inputs reference valid unspent outputs
  • Contribution Transaction Parsing: Extracts and validates contribution data from OP_RETURN outputs
  • Double-Spend Prevention: Checks mempool for conflicting transactions

Block Validation

Blocks are validated through multiple stages:

  • Proof-of-Work Verification: Validates Scrypt PoW using CheckAuxPowProofOfWork() or CheckProofOfWork()
  • Merkle Root Validation: Verifies block's merkle root matches transaction tree
  • Transaction Ordering: Ensures coinbase transaction is first
  • MWEB Extension Block Validation: Validates MWEB extension block structure and transactions
  • Size Limits: Validates block size does not exceed 1 MB

Genesis Block Validation

The genesis block (block 0) is treated specially during validation:

Genesis Block Mining

During chain parameter initialization (CRegTestParams constructor), the genesis block is explicitly mined with valid Proof of Work:

// From chainparams.cpp (regtest)
genesis = CreateGenesisBlock(1704067200, 0, 0x207fffff, 1, 10000 * COIN);

// Mine genesis block with valid PoW
while (!CheckProofOfWork(genesis.GetPoWHash(), genesis.nBits, consensusParams)) {
    ++genesis.nNonce;
    if (genesis.nNonce == 0) {
        genesis.nTime++;
        genesis.nNonce = 0;
    }
}

// Verify mined genesis block
assert(CheckProofOfWork(genesis.GetPoWHash(), genesis.nBits, consensusParams));
consensus.hashGenesisBlock = genesis.GetHash();

Genesis Block Validation Flow

When reading blocks from disk (ReadBlockFromDisk in validation.cpp):

  1. Always Validate PoW: All blocks, including genesis, must pass PoW validation
  2. AuxPow Handling: CheckAuxPowProofOfWork() is called for all blocks
  3. Genesis Fallback: Since genesis blocks don't have AuxPow, the function falls back to CheckProofOfWork()
  4. Guaranteed Validity: Because genesis block is mined during initialization, it always passes PoW validation
  5. Checkpoint Verification: Genesis block hash is stored in checkpoint data at height 0

Validation Functions

  • CheckProofOfWork(): Validates Scrypt PoW hash against target difficulty
  • CheckAuxPowProofOfWork(): Validates AuxPow for merge-mining (falls back to CheckProofOfWork() for genesis)
  • AcceptBlock(): Main block acceptance function that validates PoW during initial acceptance
  • ReadBlockFromDisk(): Reads blocks from disk and validates PoW

Block Reading and Validation

ReadBlockFromDisk() in validation.cpp ensures all blocks are validated:

// Check the header - always validate PoW for all blocks
// Genesis block PoW is validated during AcceptBlock, but we still check here for consistency
if (fCheckPOW && !CheckAuxPowProofOfWork(block, consensusParams))
    return error("ReadBlockFromDisk: Errors in block header");
Critical: The genesis block is explicitly mined with valid PoW during chainparams.cpp initialization. This guarantees that CheckProofOfWork() will always pass for the genesis block, ensuring blockchain integrity from block 0.
